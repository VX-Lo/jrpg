## Project rules (never violate)
1. **Determinism** ‚Äî same seed + inputs = bit-identical log. Tested, not assumed.
2. **Generator, not database** ‚Äî derivable from seed ‚Üí not stored. The save file is seed + ordered event log.
3. **Meta-fence** ‚Äî the meta-profile modifies the party's *starting state* and nothing else. It is **never** an input to worldgen. `worldgen(seed, tierIndex) ‚Üí Tier`, exactly 2 params, enforced by an arity tripwire.
4. **Oracles logged, never recomputed.**
5. **Engine package: ZERO DOM, ZERO React, no `Math.random`/`Date.now`/ambient state.** Enforced by tsconfig lib-omission + eslint `no-restricted-*`, not discipline.

## Architecture map
Monorepo, npm workspaces, root `"type": "module"`. `packages/engine` is the pure-TS headless engine (Node-testable only); a thin renderer will read state and draw it (Phase 10).

- `src/rng/` ‚Äî FNV-1a 64-bit hash, SplitMix64 `Rng` + `createRng`. `substream(key)` derives from the stream's **immutable seed**, not its draw cursor (Invariant #1). Keys hashed directly into a seed; no sequential derivation.
- `src/log/` ‚Äî append-only `Event {seq, tick, type, payload}`. `seq` monotonic at append; `tick` is the universal clock driving all downstream time derivation. BigInt-safe serializer.
- `src/oracle/` ‚Äî `OraclePort<In,Out>`; live logs `{input,output}`, replay reads back and throws past what was logged (never silently falls back to live).
- `src/harness/` ‚Äî `GameState {seed,rng,log,tick,tier,partySpec}`; `constructState` calls real `worldgen()`. CLI: `generate`/`replay`/`diff`/`gen`/`quest`/`embed`. **This is how deep-tier balance work happens ‚Äî runs are too long to play there.** (`partySpec` still a stub ‚Üí Phase 5.)
- `src/worldgen/` ‚Äî Phase 2. `worldgen(seed, tierIndex)`, fenced to 2 params. `band(tierIndex) = round(BAND_BASE ¬∑ tierIndex^BAND_GROWTH)`, `BAND_GROWTH = 1.4` (must stay > 1 ‚Äî the number the whole meta-curve rests on). 4 fixture kernels (Fen/Cinderreach/Hollow/Canopy). ## Invariants worth not breaking
Subtle rules where a "reasonable" change reintroduces a solved bug. Read before touching the named area.

1. **RNG** ‚Äî `substream` derives from the immutable seed, not the mutable draw cursor. This is *why* determinism is independent of call count/order. `nextInt` uses rejection sampling (unbiased; consumes a seed-determined variable number of draws).
2. **Substream key granularity** ‚Äî per-cell/per-tile where placement must be order-independent (`spatial:‚Ä¶:chunks:{cx},{cy}`, `:fill:‚Ä¶`); tier-scoped where the constraint is tier-global (Voronoi seeds; kernel assignment ‚Äî without-replacement so no two regions share a kernel). **Never per-region for an order-sensitive draw** ‚Äî a later scan-order change would silently reshape every region.
3. **Voronoi** (`voronoi.ts`) ‚Äî seed placement is one tier-scoped draw. Which region occupies which seat is chosen by **scoring all ‚â§24 candidate layouts against the adjacency graph** (no RNG), which is what stops physically-unreachable regions. The **territory-floor repair** claims `UNOWNED_CELL`s first, **never a donor's own seed cell** (stealing it makes `enforceContiguity` disown the whole blob ‚Üí severs regions), fixes regions in adjacency-fair order, and **iterates repair‚Üîcontiguity to a fixed point** (`TERRITORY_REPAIR_ROUNDS`) since each pass could otherwise undo the other. *(This is the fix for the softlock + territory-floor crashes found beyond CI's old sampled range.)* White noise at strength 5.0 beat box-blurred (measured); don't re-try blurred.
4. **Assembly** (`assemble.ts`) ‚Äî single row-major pass, **no backtracking, no WFC** ‚Äî safe *only because* the fixture library covers all 16 open-edge patterns (asserted). Don't add propagation; preserve the coverage invariant. Unfillable cells throw as fixture-coverage gaps.
5. **Off-backbone cells** ‚Äî exactly two honest options: linked-to-backbone or sealed-on-all-sides. Never walkable islands (Gate 3). Enclaves forbidden adjacent to logical-node cells.
6. **Entry‚ÜíPOI spine** carved first and protected from sealing, at every scale (backbone-first, one scale down).
7. **POI chunks** only ever land on logical-node cells (symmetric filter ‚Äî a non-node cell may not take a POI chunk).
8. **decor/fill** writes a **separate layer** that can never flip a walkable tile impassable ‚Äî the backbone can't be severed after it's guaranteed.
9. **`embedTier`'s 3rd param** injects the chunk **library** (authored content) only ‚Äî never party/meta/run-state. Not a meta-fence loophole.
10. **Solver A** (`solverA.ts`) ‚Äî **VISITING** (reads hints, always free) is strictly separate from **INTERACTING** (checks a node for a fact ‚Äî legal only when candidates narrowed to one, or as a logged guess). Conflating them produced a false zero-guess signal (hints weren't load-bearing). **The single most important solver invariant.**
11. **Quest structure** ‚Äî regions build independent local DAGs; **only the boss region's chain is load-bearing** (Solver A scoped to it). Solver B still verifies the whole tier structurally. **Locks gate a node's *usability*, never physical traversal** (region graphs are already fully connected).
12. **`band()`** is deliberately float (derived, never stored) ‚Äî don't "fix" it to integer-only.
13. **`MultiplierKind`** intentionally includes the 3 forbidden unbounded kinds (`job_level`/`primary_stat`/`weapon_scaling`) so the loader has something concrete to reject (Gate 4). `BOUNDED_MULTIPLIER_KINDS` (`constant`/`crit`/`elemental`) is the real allow-list.
14. **`Ability.primaryDomainTag`** = "this ability's main specialty," **any** tag category (not restricted to domain tags) ‚Äî a martial class needs mastery on its martial signature. Don't re-tighten.
15. **Content layout** ‚Äî single array-of-tables files for `tags`/`weapons`/`threatArchetypes`/`weaknesses`; **one file per entity** for `jobs`/`abilities`/`grammars` (hot-load = drop a file).
16. **Tests that mutate on-disk content** copy the content dir into an OS temp dir per test ‚Äî `loadContentFromDir` scans the whole dir, so writing into the shared dir contaminates parallel vitest files.
17. **Settled tooling, don't re-litigate:** npm workspaces; vitest; `smol-toml` (runtime dep); Node 20 in CI; FNV-1a + SplitMix64 core.
18. **Rule 10 at the registry** (`battle/registry.ts`) ‚Äî a multiplicative modifier on an unbounded variable (the 5 primary stats) throws `Rule10Violation` at `addModifier`, not just at content-load time. Every bounded variable has a declared clamp in `VARIABLE_CLAMPS`, re-applied on every `resolveVariable` call ‚Äî stacking can never push a bounded variable past its range. This is Phase 3's rule 10 extended to runtime modifiers, the same rule, one more enforcement point.
19. **No `canEngage`** ‚Äî searched for and asserted absent by Gate 6. The soft gate (counter-material) is two scalars only (a stat edge + variance widening), never a boolean that blocks initiating a fight. Don't reintroduce one, even as a "balance" convenience.
20. **Accuracy resolves before the weakness/Break check, always** (`damage.ts`) ‚Äî a documented, stable order so a simultaneous glancing-blow scalar and a Break trigger never race (CLAUDE.md Phase 5 ¬ß13.4). Don't reorder without re-reading that section.
21. **Party turn inputs are consumed strictly in queue order** ‚Äî `runBattle` throws if the next `BattleInput`'s `actorId` doesn't match whoever the CTB clock actually picked next. This is deliberate: the determinism contract is "ordered `BattleInput[]` matching queue order exactly," not "inputs tagged by actor, reordered by the engine." A caller (future UI) must compute whose turn is next before prompting.
22. **Weapon-reach targeting is auto-redirected, never rejected** ‚Äî if a melee attacker's requested target is in a non-empty front row's shadow (illegal per D6), the resolver silently redirects to the front-row-eligible target rather than throwing, consistent with the auto-retarget "never a wasted turn" philosophy (CLAUDE.md ¬ß9).

## Open questions

Judged across many playthroughs (does reaching tier N+1 feel like the right run-to-run improvement), not answerable from a spec session. 2. **Traversal trivialization ‚Äî MEASURED.** Backbone tick cost is **flat** across tiers (mean ~123 ticks; tier-1 127.2 vs tier-100 124.8; linear slope 0.057 ticks/tier, negligible vs per-tier œÉ 40‚Äì57). Consistent with `NODES_PER_TIER` fixed at 24 ‚Äî only the difficulty band grows with depth, not world size. So the asymmetry is **real**: at high meta, combat trivializes (auto-resolve ‚Üí microseconds) but traversal does not. **Decision pending** (not blocking): accept it (exploration has value regardless of combat difficulty ‚Äî Rimworld/Kenshi don't make traversal free either), *or* add the one natural lever ‚Äî a **one-time, bounded, meta-gated flat reduction to far/farthest-zoom tick cost**. Rule-10-safe: flat, bounded, a discrete unlock (not a scaling primary, not a talent tree).

2. **`channel` on `DamageParams`** (the other ASK-FIRST item) ‚Äî added additively as `channel?: DamageChannel`, defaulted at validation time (`weapon_scaling` base ‚Üí `physical`, else `magical`). All existing Phase 3 content/gates stayed green untouched ‚Äî confirmed this is the sanctioned exception to "nothing upstream imports `battle/`."
3. **No encounter-builder wiring to live worldgen/questgraph state.** The spec allows `archetype + band ‚Üí combatant` to live in `battle/` as a separate module; this session didn't build it because nothing in `questgraph/`/`spatial/` assigns a concrete tier band + archetype to an *encounter* yet (only `ThreatArchetypeId` references on quest facts). `BattleRequest` is constructed directly (by tests, and eventually by whatever Phase 6/7 code first needs a live encounter). Revisit when that caller exists ‚Äî don't build the wiring speculatively.
4. **Enemy break shield derivation deferred with it** ‚Äî same reason as #3: `EnemyCombatantRequest.breakShieldMax` is provided by the caller, not derived from `ThreatArchetype.powerTier` by an engine formula, since there's no live caller yet to get the formula right for.
5. **Boost is party-only.** Enemies never accrue or spend Boost ‚Äî it's described in the spec as the player's burst-timing economy; the MVP `ai.ts` priority list has no Boost-spend decision, so wiring boost onto enemies would be inert code. Revisit if/when enemy Boost-spend AI is ever wanted.
6. **`battle:dot` and `battle:break` log event types added** beyond the spec's explicit list, during the Gate 7 spot-check, because DoT ticks and Break triggers were otherwise invisible in the raw log (only inferable from HP deltas) ‚Äî failed the "reads as a legible battle with no UI" bar on first attempt. Structural-only addition, no semantic engine change.
7. **Defend's "reduces next action cost"** implemented as a one-shot `nextActionCostMult` (0.75) applied the next time the guarding actor's own turn comes up, then consumed ‚Äî not a registry modifier, since its duration is "until next action" rather than a tick count known in advance.
8. **`shift_queue` direction semantics**: `"back"` delays (increases `nextActionTick`), `"forward"` advances (decreases it, floored at the current tick so the queue can't be pushed into the past).
9. **Target-side resolution has no offensive/defensive ability classification.** The resolved target's own side (party vs enemy) ‚Äî not the ability's primitive mix ‚Äî determines the target set for `row`/`all` shapes. This sidesteps needing to classify abilities as beneficial/harmful in data; a heal ability just needs `targetId` pointed at an ally. Weapon-reach geometry (front-row-unless-empty) is applied only when the ability contains a `damage` effect and targets the opposing side, so it never constrains a healer targeting their own back-row ally.

### Decisions log
1. **Tab switch: number keys `1`/`2`/`3`** (Map/Content/Battle). Persistent footer shows the active tab, current seed, and a context-sensitive keybinding legend per tab ‚Äî Lo isn't expected to remember hotkeys between sessions.
2. **Map zoom: single cycle key `z`**, order far ‚Üí medium ‚Üí closest ‚Üí farthest ‚Üí far. `medium` shows a fixed "no chunk library yet" message instead of touching any spatial code (nothing downstream is actually invoked, mirroring `cli.ts`'s own `cmdEmbed` handling). `farthest` renders `farthestZoomView(tier)` as a plain node/edge text list ‚Äî no grid, no cursor, matching "pure abstraction over the logical graph" from `spatial/zoom.ts`. Movement is arrows or WASD, real walkability (`isWalkableCode` against the assembled `TileMask`), blocked tiles are a no-op with a status line, never fly-through.
3. **Map regenerate (`r`) / teleport (`t`) / inspect (`i`)** are dedicated keys with inline text prompts (own local mode, not a separate screen). Teleport accepts `"x,y"` tile coordinates or (far zoom only) a logical node id looked up against `spatial.placements`. Inspect is read-only: cell ‚Üí `PlacedChunk` ‚Üí chunk id/biome (via `defaultChunkLibrary()`)/backbone flag/hosted `WorldNode` (id, kind, name, `canHostQuestNode`, `isBoss`) if any.
4. **Content tab covers all 5 listed categories** (jobs, abilities, weapons, threat archetypes, tags) ‚Äî the task spec's prose said "sub-navigating the four content categories" while listing five; resolved by including everything actually listed rather than dropping one arbitrarily. Left/right (or h/l) cycles category, up/down navigates, Enter opens detail, Esc/Backspace returns. Detail view is a raw `JSON.stringify` of the entity ‚Äî deliberately no bespoke per-type renderer, so it can never drift out of sync with `content/types.ts`.
5. **Interactive battle turn-by-turn, via one additive engine export.** `runBattle` takes a full ordered `BattleInput[]` and always ran to completion or threw ‚Äî no incremental/resumable API existed, and CLAUDE.md invariant #21 already anticipated this ("a caller must compute whose turn is next before prompting"). Added `AwaitingPartyInputError` in `packages/engine/src/battle/resolver.ts` (barrel-exported from `battle/index.ts`): thrown, carrying the live `party`/`enemies`/`actorId`/`tick` at the exact pause point, **only** when the scripted input stream runs dry ‚Äî a mismatched `actorId` (a real caller bug) still throws a plain `Error` exactly as before. `debug-tui` replays `runBattle(request, inputs, ctx)` from tick 0 with a **fresh** rng substream and a fresh `EventLogWriter` every time the committed `inputs[]` grows by one input (cheap ‚Äî fights are short and headless) and renders straight off the caught error's live state. No resolver logic is duplicated; the ability/target menu commits exactly the `BattleInput` shapes any scripted caller would. This is the additive, non-breaking kind of engine change the task brief pre-authorized ‚Äî existing tests, gates, and CI cover it unchanged (confirmed: no test asserted on the old "end of input" error message).
6. **¬ß5 gap ‚Äî `PartySpec`:** still a stub in `packages/engine/src/harness` (`memberNames` only, unchanged). Built the minimal debug-only assembler in `debug-tui/src/engine/party.ts` (`debugBuildPartyMember`/`debugPresetParty`), **not** the real Phase 11 run-lifecycle party system (recruitment, persistence, chronicle callback are separate, later work). Default preset: `job:warrior` + `job:mage` at level 5; abilities = signature abilities ‚à™ tag-eligible abilities (`canEquip`), capped at `EQUIP_CAP`; HP/MP via explicitly-labeled UNCALIBRATED placeholder formulas (no real curve exists yet either).
7. **¬ß5 gap ‚Äî enemy stats + gear roll:** no archetype+band ‚Üí `EnemyCombatantRequest` formula and no gear-instantiation function exist anywhere in `packages/engine` (Phase 5 Decisions #3/#4 deferred the former for lack of a live caller; the latter is Phase 13 crafting). Built `debugBuildEnemy`/`debugRollEncounter` in `debug-tui/src/engine/encounter.ts` and `debugRollGear` in `debug-tui/src/engine/gear.ts` ‚Äî small, explicitly-labeled, UNCALIBRATED placeholder formulas, named and located so neither is ever mistaken for real Phase 6/7/13 work later. **Scope trim:** manual per-archetype encounter picking (the spec's first-offered option) was not built for v1 ‚Äî only "roll a random encounter at a chosen tier band" (the spec's explicit alternative), adjustable via `+`/`-` before a fight starts (locked once the first input is committed, so a live fight's already-targeted enemy ids can't go stale).
8. **Reset (`r`) vs reroll encounter (`n`) vs reroll rewards (`g`, post-victory only).** Reset clears `inputs` and replays the *same* `BattleRequest`/`encounterId` from scratch ‚Äî bit-identical up to whatever new inputs follow. Reroll encounter advances both an encounter-roll rng substream counter and the fight's own `encounterId` sequence number, so a rerolled fight's rng draws never collide with the abandoned one's. Reroll rewards re-invokes only `debugRollGear` under its own substream key (`debug-tui:reward-roll:{encounterId}:{seq}`) ‚Äî it never re-simulates combat, so Lo can spam it to eyeball gear variety.
9. **All debug-only rng draws are substreamed under `"debug-tui:"`** (encounter rolls, reward rolls) so they can never collide with a real engine substream key (RNG invariant #2) ‚Äî this tool has no real state, but it still isn't exempt from using substreams correctly.
