## Project rules (never violate)
1. Determinism: same seed + inputs = bit-identical log. Tested, not assumed.
2. Generator, not database: derivable from seed → not stored.
3. Oracles logged, never recomputed.
4. Engine package: ZERO DOM, ZERO React. Enforced by lint/boundary.
5. No Math.random(), no Date.now(), no ambient state in the engine. Ever.

## Architecture
- Monorepo root uses npm workspaces (no extra tooling needed at this size). Root `package.json` has `"type": "module"`.
- `packages/engine` — pure TypeScript engine package. No DOM lib, no React. Node-testable only.
  - `src/rng/` — `fnv1a.ts` (FNV-1a 64-bit hash), `splitmix64.ts` (`Rng` class + `createRng`), `index.ts` (barrel)
  - `src/log/` — `types.ts` (`Event`), `writer.ts` (`EventLogWriter`), `reader.ts` (`EventLogReader`), `replayer.ts` (`replay`), `serializer.ts` (`serializeLog`/`deserializeLog`, BigInt-safe), `index.ts`
  - `src/oracle/` — `types.ts` (`OraclePort`), `liveOracle.ts`, `replayOracle.ts`, `mockOracle.ts` (`createCountingOracleSource`), `index.ts`
  - `src/harness/` — `types.ts` (`GameState`/`TierSpec`/`PartySpec` stubs), `state.ts` (`constructState`), `demoScenario.ts` (Phase-1-only fixture, NOT game content), `cli.ts` (generate/replay/diff/gen/quest), `prettyPrintTier.ts`, `prettyPrintQuestGraph.ts`, `index.ts`
  - `src/questgraph/` — Phase 4. `types.ts` (Hint vocabulary, `QuestFact`/`QuestLock`/`RegionQuestGraph`/`TierQuestGraph`), `dag.ts` (`buildRegionDag` — forward generation), `hints.ts` (`emitRegionHints`, `factDisplayId`), `mercy.ts` (`applyMercyPass`, `computeLiveHintedFactIds`), `solverA.ts`/`solverB.ts` (the two-solver gate), `solverShared.ts`, `reachability.ts` (`reachableAtStep`/`unlockedAtStep`), `validate.ts`, `serialize.ts`, `questgraph.ts` (orchestrator: `buildTierQuestGraph`), `arcs/` (`types.ts`, `templates.ts` — 4 fixtures, `select.ts`, `arcEngine.ts` — `simulateArc`/`logArcTerminalEvent`, `tierArcs.ts` — `buildTierArcs`), `index.ts`. See the Phase 4 architecture section below for the full design.
  - `test/` — vitest suites: `rng.test.ts`, `log.test.ts`, `gate1.replay.test.ts`, `gate2.perturbation.test.ts`, `gate3.oracle.test.ts`, `gate5.boundary.test.ts` (Gate 4 covered inside `log.test.ts`'s round-trip tests), plus `test/worldgen/`, `test/content/`, `test/questgraph/` per-phase gate suites
- `.github/workflows/ci.yml` — runs `eslint .`, `npm run build --workspace packages/engine` (tsc), `npm run test --workspace packages/engine` (vitest) on push to main and on PRs. Uses Node 20 (local dev used Node 18.20.8 — works, but some devDependency engines warn below 20; CI pins 20 to be clean).
- Substream key convention: `domain:scope:id:purpose`, e.g. `worldgen:tier:5:regions`. Keys are hashed directly (via FNV-1a 64-bit) into a SplitMix64 seed — no sequential derivation, ever.
- Hash function: FNV-1a, 64-bit, integer-only (BigInt), operating on UTF-8 bytes. A few lines of pure integer arithmetic with a fixed, documented spec — no platform-dependent float or crypto-library behavior. `deriveSeed(parentSeed, key)` hashes `u64ToBytes(parentSeed) || utf8(key)`.
- PRNG core: SplitMix64 (BigInt arithmetic only, masked to 64 bits throughout). Each `Rng` instance stores its own immutable `seed` separately from its mutable advancing `state` — `substream(key)` derives the child from `this.seed` (never from wherever the draw cursor happens to be), which is *why* Gate 2 holds: independent of call count and call order.
- `createRng(seed)` accepts `bigint | number | string` — numbers/bigints seed directly (masked to 64 bits), strings are hashed via `deriveSeed(0n, seed)`.
- `nextInt(bound)` uses rejection sampling against 2^64 (not modulo) for unbiased output — still fully deterministic, just consumes a variable, seed-determined number of draws.
- Event log shape: `{ seq: number, tick: number, type: string, payload: unknown }`. `seq` assigned monotonically at append time by `EventLogWriter`; `tick` is the universal game clock — drives all downstream time-based derivation (faction updates, market drift, event rolls) starting Phase 2+. Log is append-only; nothing is ever mutated or removed post-append.
- Serializer tags BigInt values as `{ __bigint__: "123" }` via a JSON replacer/reviver, since payloads legitimately contain BigInts (seeds, 64-bit draws) and plain `JSON.stringify` throws on them.
- Ports defined so far: `OraclePort<In, Out>` with `query(input): Out`. `LiveOracle` invokes the source and appends `{input, output}` under a caller-chosen event `type`; `ReplayOracle` reads events of that same type back in order and throws (does not silently fall back to live) if queried past what was logged.
- `GameState` shape: `{ seed, rng, log, tick, tier, partySpec }`. `tier` is now the REAL generated `Tier` (Phase 2's worldgen output), not a stub — `constructState(seed, tierSpec, partySpec)` calls `worldgen(seed, tierSpec.tierIndex)` internally. `TierSpec` is now just the input `{ tierIndex: number }`.
- CLI (`packages/engine` → `npm run cli -- <command>` or `npx tsx src/harness/cli.ts <command>`):
  - `generate <seed> [outFile]` — constructs state (tier 1), draws from a named substream, writes/prints the serialized log (Phase 1's demo scenario was deleted and replaced with an inline substream draw once real worldgen existed — see Decisions)
  - `replay <logFile>` — deserializes a log and folds it into a trivial `{eventCount, lastTick}` summary via `replay()`
  - `diff <fileA> <fileB>` — byte-compares two serialized logs, reports first differing offset, exit code 0/1
  - `gen --seed X --tier N [--print]` — Phase 2. Calls `worldgen(seed, tierIndex)` directly; `--print` renders `prettyPrintTier()` (human-readable), otherwise dumps `serializeTier()` JSON. This is how Gate 6 and all future balance work get done — runs are too long to play to deep tiers.

## Phase 2 architecture — world generator (`packages/engine/src/worldgen/`)
- **THE FENCE:** `worldgen(seed: bigint|number|string, tierIndex: number): Tier` — exactly 2 parameters, enforced structurally (nothing downstream receives party/meta/run-state) and by an arity tripwire test (`worldgen.length === 2`) in `test/worldgen/gate2.colddepth.test.ts`.
- `config.ts` — every tunable constant, documented, several marked UNCALIBRATED. `BAND_GROWTH = 1.4` (must stay > 1, see its doc comment — this is the single number the whole meta-progression system rests on).
- `types.ts` — `Tier`, `Region`, `WorldNode`, `WorldEdge`, `DungeonCluster`, `Faction`, `Npc`, `RegionKernel`, `ThreatArchetypeRef`, `EconomicAxis`.
- `band.ts` — `band(tierIndex) = round(BAND_BASE * tierIndex^BAND_GROWTH)`. Pure function of tierIndex only. Uses `Math.pow`/`Math.round` (float) deliberately — unlike the RNG/hash core, band() is never stored (rule #2: derivable from seed, recomputed every time) so it doesn't need cross-platform-forever bit-stability, only purity within one process. Documented in the code.
- `kernels/` — 4 fixture `RegionKernel`s (Fen, Cinderreach, Hollow, Canopy), each with lexicon roots, 1-2 boundary roots shared with a ring-adjacent neighbor, an economic axis, a puzzle-affinity tag, an arc-pool tag, and curated (hand-written, not grammar-concatenated) name pools of ~12 entries each for settlements/npcs/factions/landmarks. Full 15-25 kernel set is Phase 3.
- `kernelAssignment.ts` — **deviates from the suggested per-region substream key.** Kernel exclusivity (no two regions in one tier share a kernel — needed since only 4 fixture kernels exist and a collision would produce duplicate proper nouns, which Gate 1 forbids) is a tier-scoped fact, so it's one coordinating draw at `worldgen:tier:{n}:regions:kernels` (a shuffle-and-take, without replacement) rather than independent per-region draws. Documented in the code; flagged to the user as a judgment call, not blocked on.
- `nodes.ts` — node-kind budget per region (`computeNodeKindCounts`, fractions from config, settlement count floored at 1), node 0 always forced to be a settlement (the region's entry point), names drawn without replacement from the kernel's curated pools.
- `edges.ts` — weighted region graph: random spanning tree (guarantees full reachability from node 0) + extra edges for cycles/texture. Landmark-endpoint edges draw from a wider weight range (`EDGE_WEIGHT_REMOTE_MAX_TICKS`) — this is the whole mechanism for "high peaks are slow," no special-casing beyond node-kind-keyed range selection.
- `dungeon.ts` — multi-level node clusters hanging off `dungeonEntrance` nodes. NOT counted toward the region's 24-node-tier budget (an appendage, not part of the region's own graph). Depth-node names derive from the (already tier-unique) parent name, so no separate curated pool is needed.
- `factions.ts` / `npcs.ts` — factions are name + home region + `reputation: 0`, nothing else (no territory/diplomacy/simulation — that's Phase 7's fence too). NPCs: curated name + origin region + one trait + one job, drawn from small generic fixture trait/job pools (not proper nouns, no uniqueness requirement).
- `boss.ts` — exactly one boss per tier: picks a region, then a landmark node within it (never a settlement — every region always has ≥1 landmark by construction), marks `isBoss: true`, attaches a `ThreatArchetypeRef` (an ID string from a 5-entry fixture list in config — Phase 5 implements the mechanical guts). No quest chain built to it (Phase 4's job).
- `worldgen.ts` — orchestrator. Builds one root `Rng` from `seed`, threads it through `rollShape` → `assignKernels` → `generateRegion` (×N) → `placeBoss`/`applyBoss`. A fresh root is constructed every call — nothing cached across calls, which is what makes Gate 2 hold.
- `serialize.ts` — `serializeTier(tier)`, a BigInt-safe `JSON.stringify` for byte-identical comparisons in gates.
- Substream keys actually used (deviates from the prompt's suggested list only in `regions:kernels`, see above): `worldgen:tier:{n}:shape`, `worldgen:tier:{n}:regions:kernels`, `worldgen:tier:{n}:region:{i}:nodes`, `worldgen:tier:{n}:region:{i}:edges`, `worldgen:tier:{n}:region:{i}:dungeon:{entranceNodeId}:cluster`, `worldgen:tier:{n}:region:{i}:factions`, `worldgen:tier:{n}:region:{i}:npcs`, `worldgen:tier:{n}:boss`. No `worldgen:tier:{n}:band` key — band is a pure formula, nothing to draw.

Additive between stat and job-level; multiplicative only by bounded things — rule 10, executable.

## Phase 4 architecture — quest graph (`packages/engine/src/questgraph/`)

This phase's prompt specified the hint vocabulary, the forward-generation rule, the two-solver gate, mercy pinning, and the arc Beat mechanism concretely, but left several structural questions genuinely open (the prompt itself flagged three as ask-first — none of the below are those three — everything else was "otherwise, proceed" territory, judgment calls made and documented here, same discipline as Phase 2's kernel-assignment deviation).

**THE CENTRAL ARCHITECTURE DECISION: regions build independent local quest DAGs; only the boss region's chain is load-bearing for tier completion.** Each region (`RegionQuestGraph`) gets its own forward-built gate sequence, facts, hints, skill trainers, and elite captures — fully self-contained, matching the given per-region substream keys (`questgraph:tier:{n}:region:{i}:dag/items/skills/hints/mercy`). The tier's boss node lives in exactly one region; in that region only, the LAST gate in the sequence is forced onto the boss node itself (never moved, never redefined — Phase 2's placement stands). Every other region's chain gates its own optional side content (skill trainers, elite captures, flavor items) — real, structurally sound (proven by Solver B / Gate 1, which runs tier-wide), but not part of the tier's actual win condition. **Solver A is scoped to the boss region only** (see below) — this is what "tier N's quest graph is completable by inference" (ROADMAP.md §7) cashes out to under this design, and it keeps Gate 2's guessCount/tedium metrics meaningful (a guess is always a real hole in the golden path's hint economy, never noise from wandering an unrelated region).

**Locks gate a node's USABILITY, never physical traversal.** Phase 2's region graphs are already guaranteed fully connected (spanning tree + extra edges, see `worldgen/edges.ts`) — there is no geometric cutting to do without articulation-point analysis this phase doesn't need. A "locked node" can always be walked up to (revealing its lock's required fact ids — the keyhole-shape rule; otherwise a fact with a thin hint economy could be permanently undiscoverable, which would be a generation bug, not a solver bug), but its content (item pickup, trainer, capture, boss encounter) is unusable until its lock's required facts are all acquired. `questgraph/reachability.ts`'s `reachableAtStep(allNodeIds, gateNodeIds, step)` is the whole mechanism: a node is excluded once its position in the ordered gate sequence is `>= step`.

**Forward generation, structurally enforced (Deliverable 2's core requirement, and the thing Gate 1 exists to prove held).** `questgraph/dag.ts`'s `buildRegionDag` picks the region's ordered gate sequence FIRST (2-4 nodes drawn from `canHostQuestNode` candidates, entry node and boss node never mid-sequence; boss region forces the boss node as the final gate). Then, for gate `i`, its fact(s) are drawn ONLY from `reachableAtStep(i)` — nodes not reserved by gate `i` or any later gate. This makes deadlocks structurally impossible: nothing gate `i` needs can ever be locked behind gate `i` or anything after it. No validator runs after the fact — there's nothing to catch, by construction. Gate 1 (1000/1000 seeds, Solver B) confirms this empirically rather than trusting the argument blindly.

**Multi-path and compound locks are both real, not rare edge cases.** `QUESTGRAPH_MULTI_PATH_CHANCE` (0.4) governs whether a fact is placed at 1 or 2 reachable nodes (either independently supplies it — bad-luck protection). `QUESTGRAPH_COMPOUND_LOCK_CHANCE` (0.3) governs whether a lock requires 1 or 2 facts (AND) — the "the Key needs the Rod and the Orb" case `DependencyHintFact` narrates.

**A gate's prerequisite can be an item, a skill, or an elite capture — not only items.** `pickFactKind` weights item:skill:eliteCapture roughly 70:15:15. This reads Deliverable 2's parenthetical ("place a prerequisite (item, skill trainer, or elite-capture node)") as meaning a LOCK can genuinely require proof of a captured elite or a learned skill, not only a fetched item — distinct from Deliverable 8's SEPARATE, NON-GATING skill-trainer/elite-capture placements (below), which reuse the identical `reachableAtStep` mechanism but never feed a lock.

**Deliverable 8, "not a parallel system," implemented literally**: `SkillTrainerPlacement`/`EliteCapturePlacement` are drawn from `reachableAtStep(allNodeIds, gateNodeIds, step)` for a randomly chosen step, 0-2 per region each (`SKILL_TRAINERS_PER_REGION_*`/`ELITE_CAPTURES_PER_REGION_*`), sharing the region's `:skills` substream (no separate `:elites` key was given in the suggested list; documented deviation, not blocking). They never gate anything — Gate 6 verifies reachability via `computeReachabilityFixpoint` (a full-tier fixpoint, not just the boss lock) rather than via a solver, since "no orphans" doesn't require inference, just structural proof.

**"Common-tier" abilities are derived, not newly schema'd.** `questgraph/questgraph.ts`'s `computeCommonAbilityIds` returns every `Ability` not listed in any `Job.signatureAbilities` — reuses Phase 3's existing signature/non-signature split exactly, no new field. Regional-scope `ThreatArchetype`s (`scope: "regional"`) are the elite-capture pool, falling back to world-scope if a fixture set has none.

**Hint display ids are always fact-unique, even for skill/eliteCapture facts** (`questgraph/hints.ts`'s `factDisplayId`: `${abilityId}#${factId}` / `${threatArchetypeId}#${factId}`, vs. items' already-unique `item:${factId}`). Caught during implementation: with only 4 Common abilities and 1 regional threat archetype in the fixture pool, the SAME content id is very likely picked by more than one fact across a tier (a second Weaken trainer, a second regional ambusher) — narratively harmless, but a bare content id would make a hint referencing it genuinely ambiguous about which lock it resolves. Fact-suffixing sidesteps this without requiring cross-region coordination to dedupe content ids at generation time.

**Hint emission** (`questgraph/hints.ts`): for each fact, `HINTS_PER_FACT` (3) hints are sourced from `reachableAtStep(fact.gateStep)` (excluding the fact's own placement nodes where possible), ranked by `unlockedAtStep` (how recently that node became free relative to the gate) and bucketed across `HINT_SPECIFICITY_CURVE` (vague→medium→precise) — earlier-freed nodes get vaguer hints, later-freed nodes get precise ones. `LocationHint`/`ExclusionHint` operate at REGION granularity and may reference OTHER tier regions as decoys (drawn from `allRegionIds`, always excluding/including the true region correctly so no hint is ever untruthful) — this is what gives `candidateRegions`/`region` real narrowing content without requiring hint SOURCES to live outside the fact's own region (they never do — matches the per-region substream key). `ProximityHint` always names a true placement node, at any specificity it's chosen for. One `DependencyHint` is emitted per compound lock, foreshadowing the AND relationship ahead of encountering the lock itself — mechanically near-redundant with the structural lock-reveal (see below) but real legibility value and required by the vocabulary.

**Solver A / B correctness note — a real bug caught before shipping.** An early draft of Solver A auto-resolved every known fact against every visited node regardless of hint narrowing (since "visiting" was unconditionally free and `reconcile()` checked ground truth automatically). The tell: 300 test seeds solved with a guess count of exactly zero — too good to be true, since it meant hints weren't actually load-bearing, just color. Fixed by strictly separating VISITING (reads hints at a node, always free, never touches fact content) from INTERACTING (checks whether a specific node holds a specific fact — only legal when the candidate set has narrowed to exactly that node, or as a logged guess when forward progress requires a forced choice among multiple remaining candidates and no unvisited/unencountered node remains). Post-fix: 1000/1000 seeds solved, avg 1.096 guesses/seed, avg tedium 0.319 — an honest signal. See `questgraph/solverA.ts`'s module doc comment for the full account; this is the single most important implementation lesson from this phase for anyone extending the solver later.

**Solver A's discovery model:** a fact becomes known to the solver either by reading a hint that references it, or by walking up to a locked node and learning its lock's `requiredFactIds` (translated through the same `factDisplayId` scheme) — the latter guarantees every fact is eventually discoverable even in the (rare, pool-empty) case hint emission produced zero hints for it. Region-level candidates narrow via hint intersection; node-level candidates narrow only via `ProximityHint` (exact) or via elimination (a failed guess removes that node from the candidate set). Termination is guaranteed by construction: `visited`/`satisfiedLocks` only grow, `nodeCandidates` only shrinks, all bounded by finite tier size — a generous `maxSteps` safety cap exists purely as a defensive tripwire for a future regression, not because the algorithm can legitimately hit it today.

**Mercy relocation** (`questgraph/mercy.ts`): `computeLiveHintedFactIds` pins any fact with ≥1 hint referencing it — since `HINTS_PER_FACT=3` and emission almost always succeeds when the placement pool is nonempty, nearly every naturally-generated fact ends up pinned in practice. That's the correct, intended shape (mercy adds hints; mercy never moves a hinted item), not a bug — Gate 3 is deliberately a targeted synthetic-fixture test rather than a statistical search over generated seeds, per the prompt's own framing. `applyMercyPass` takes caller-supplied `MercyPlayerStateInput[]` (no live playthrough exists yet this phase — a future live-play phase sources `ticksSinceReachable` from the real event log) and, for eligible facts crossing `MERCY_TICKS_SINCE_REACHABLE_THRESHOLD`, rolls `MERCY_ROLL_CHANCE` on a `:mercy`-suffixed substream. A hit logs a real `questgraph:mercy-relocation` event via `EventLogWriter` and collapses the fact onto one new node drawn from the same `reachableAtStep` pool placement uses (so a relocated fact can never end up behind a lock it wasn't already eligible to sit in front of).

**Regional arcs** (`questgraph/arcs/`): 4 hand-authored templates (Overthrow/political, Plague/disaster, Cult Uprising/cult, Siege/universal fallback) sharing one validated beat-graph topology (rally → escalate-or-confront → confrontation, with tick-timeout branches peeling toward StatusQuo/Lockdown along the way), re-themed per template rather than 4 unrelated shapes hand-crafted from scratch — "generated variety, authored structure" applied to arcs the same way as everywhere else. `KERNEL_ARC_THEMES` (political/disaster/cult per kernel) was reverse-engineered from Phase 2's kernel `arcPool` fixture data: each kernel's 2 authored `arcPool` entries (e.g. Fen's `["fen:smuggler-network", "fen:drowned-shrine"]`) turned out to split cleanly across exactly 2 of 3 narrative themes (political/cult/disaster) with no kernel repeating a theme and no kernel covering all 3 — a clean 3-theme × 4-kernel incidence pattern almost certainly deliberate in the original Phase 2 authoring, discovered rather than invented. `questgraph/arcs/select.ts` weights themed matches 2x over the universal fallback. No live play exists yet, so `simulateArc` (`questgraph/arcs/arcEngine.ts`) stands in for real quest-completion/tick-timeout events with one seeded roll per Beat (`ARC_BEAT_QUEST_COMPLETE_CHANCE`, 0.55) — same pattern as mercy's player-state stand-in. Terminal Beats log a real `arc:terminal` event shaped for Phase 7's Echo system (who/where/what/magnitude generated fillers over a small authored archetype, per ROADMAP.md §12) — no reference-roll or Echo-spawning logic built here, per the scope fence.

**Substream keys actually used** (deviates from the prompt's suggested list only in sharing `:skills` between trainers and elite captures — no `:elites` key was suggested and none was added):
`questgraph:tier:{n}:region:{i}:dag`, `questgraph:tier:{n}:region:{i}:items`, `questgraph:tier:{n}:region:{i}:skills` (both trainers and elite captures), `questgraph:tier:{n}:region:{i}:hints`, `questgraph:tier:{n}:region:{i}:mercy`, `arcs:tier:{n}:region:{i}:selection`, `arcs:tier:{n}:region:{i}:beats`.

Visiting (reads hints, always free) and interacting (checks a node for a specific fact, requires deduction or a logged guess) are now strictly separate. This is the single most important implementation lesson from this phase; see `questgraph/solverA.ts`'s module doc comment before modifying the solver.
- 2026-07-18 — Arc-template theme mapping (`KERNEL_ARC_THEMES` in `questgraph/arcs/templates.ts`) was reverse-engineered from Phase 2's kernel `arcPool` fixture data rather than invented fresh: each kernel's 2 authored `arcPool` entries turned out to split cleanly across exactly 2 of 3 narrative themes (political/cult/disaster), no kernel repeating a theme, no kernel covering all 3 — read as a deliberate latent structure in the original Phase 2 authoring and honored rather than ignored. The 4th template (Siege) is a universal fallback so every kernel always has ≥3 eligible templates.
- 2026-07-18 — Arc Beat transitions (quest-complete vs. tick-timeout) and mercy relocation's player-state input are both driven by a seeded roll standing in for real live-play events, since no live playthrough exists yet this phase. Both are documented as explicit stand-ins (`ARC_BEAT_QUEST_COMPLETE_CHANCE`, and `MercyPlayerStateInput` being caller-supplied rather than log-sourced) so whichever later phase wires real quest-completion/tick events and real run-state logging knows exactly what to replace.
- 2026-07-18 — Dungeon cluster interior nodes (the appendage nodes hanging off a `dungeonEntrance`, not counted toward a region's node budget per Phase 2) do NOT host quest prerequisites this phase — only top-level region nodes do. Reasoning: cluster nodes form their own linear chain reachable only through their parent entrance node, which would require recursive reachability modeling through nested clusters to place quest content in safely; out of scope for this phase's fence, noted as a stub below.
